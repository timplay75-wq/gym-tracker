# Правила разработки

## Приоритеты приложения (главное)
Gym Tracker — это инструмент для **отслеживания прогресса тренировок** и **удобного добавления упражнений и программ**. Всё остальное вторично.

**Нужно:**
- Удобный просмотр прогресса по упражнениям (графики, личные рекорды, история)
- Быстрое добавление тренировок, упражнений, программ
- Простой и понятный интерфейс без лишнего шума

**НЕ нужно (не реализовывать):**
- Достижения / бейджи / геймификация
- Социальные функции (поделиться, друзья, лайки)
- Уведомления-напоминалки и gamification-попапы
- Всё что отвлекает от основного сценария: тренироваться → записать → увидеть прогресс

## Рабочий процесс
- **Одна функция на промт** — делать одну фичу, но хорошо и полностью
- **Коммит только после подтверждения** — не коммитить, пока пользователь не сказал что всё работает
- **Всегда следовать этому файлу** — правила применяются без напоминаний
- **Как проверить** — после каждого добавления/изменения обязательно написать конкретные шаги как это проверить: какую страницу открыть, какое действие сделать, что должно появиться, какую команду запустить

## Стили / Tailwind
- Цвета писать через hex: `bg-[#9333ea]`, `bg-[#7c3aed]`, `text-[#7c3aed]` — НЕ `bg-primary-500`
- Семантические токены допустимы только если они реально объявлены в tailwind.config.js: `bg-primary-100`, `bg-primary-200`, `text-primary-600` и т.д.
- Не добавлять `dark:` классы если компонент их не использует

## Компоненты
- `Input.tsx` — всегда иметь `htmlFor` на label и `id` на input (для доступности и тестов)
- Все новые страницы/компоненты должны быть совместимы с `AuthProvider` и `ProtectedRoute`

## Тесты
- После любых изменений в компонентах — запускать `npx vitest run` и проверять что всё зелёное
- Исправлять упавшие тесты: проверять реальные классы компонента, а не предполагаемые
- `getByLabelText` работает только если есть `htmlFor`/`id` связь — иначе `getByPlaceholderText`
- `getByText` — если текст встречается в нескольких элементах, использовать `getAllByText` или более точный селектор
- `stderr` от намеренно невалидных данных в тестах — это нормально, не ошибка

## API / Auth
- Все запросы к бэкенду через `apiFetch()` из `api.ts` — JWT токен добавляется автоматически
- Токен хранится в `localStorage` под ключом `token`
- Защищённые роуты оборачивать в `<ProtectedRoute>`

## Git
- Коммитить после каждого завершённого этапа
- Формат: `feat:`, `fix:`, `test:`, `refactor:` префиксы в сообщениях

## Бэкенд
- Все роуты кроме `/api/users/register` и `/api/users/login` — защищены middleware `auth`
- Валидация входных данных через `middleware/validate.js`
- Личные записи (PersonalRecord) обновляются автоматически при завершении тренировки
